Knowledge Construction System

An agentic pipeline that builds structured, provenance-tracked brand knowledge —
so agents reason from what they've learned, not from live search at query time.
V1 · Festo PoC · May 2026

The Problem

⚡ Fragility

  • Live source calls at query time
  • APIs rate-limit, change, go down
  • PDFs parse inconsistently
  • One outage → degraded product

🕸 No Structure

  • Chunk-level RAG has no traversal
  • Can't answer: motor → VFD → compatible PLC → PLC is EOL → replacement?
  • Relationships live in PDFs, not queryable

🔁 No Learning

  • Every query re-discovers the same facts
  • No memory of what worked or failed before
  • Each new brand starts completely from zero
Thesis: Shift knowledge assembly from query time to build time. KCS pre-constructs structured, provenance-tracked artifacts that improve with every iteration.

Agents 2 & 3 · Cartographer + Schema Modeling

🗺 Samples → EntityVocabulary

How KCS Learns

Brand-level learning (per brand)  +  ALTM meta-level learning (how to learn across brands)

🔄 Run-over-run enrichment

  • Each run loads the previous KnowledgeMap + next_run_hints
  • Run 1 surveys the surface; Run 2 follows up on gaps; Run N validates
  • Failed probes are remembered — bad patterns skipped next time
  • Pagination strategies, auth patterns, cursor formats — learned once, reused forever

🧭 Meta-level learning (ALTM)

  • Learns how to learn — auth patterns, endpoint discovery, pagination strategies reused across brands
  • Similar taxonomies → seed schema hypothesis: "Festo's product hierarchy fits Siemens too"
  • Semantic retrieval of brand-agnostic knowledge to bootstrap a new brand faster

📊 Confidence-weighted knowledge

  • Every fact starts low-confidence (0.3)
  • Validated across multiple runs & sources → confidence rises
  • Contradictions from new data → flagged for HITL resolution
  • Old facts that can't be re-confirmed → decay; HITL decides

🔧 Schema evolution via graph

  • Run 1: ProductFamily modeled as flat list
  • Run 3: discovers it's actually hierarchical
  • Graph correction: rename + add -[HAS_PARENT]→ — no migration scripts
  • Relational DB would require ALTER TABLE + data migration

Open Infrastructure Challenges

🧠 Long-term memory at scale — Bedrock AgentCore Memory

V1 uses a simple TTL-based knowledge store. V2 needs managed, semantically-searchable agent memory: Amazon Bedrock AgentCore Memory offers this as a managed service — no Redis ops, native IAM, built for agent workflows. Swap point: get_store() initializer only, no agent code changes.

🌐 AGLM — graph vs. managed memory

V1 global memory is exact-key lookup. V2 needs semantic retrieval: "what schema patterns worked for similar brands?" We're evaluating Neo4j's agent memory layer (hybrid graph + vector) vs Bedrock AgentCore Memory (managed, AWS-native). Both support the three-tier model from Neo4j's research.

🔄 Data freshness / incremental sync

Strategy depends on whether the brand exposes a modified_since / changelog endpoint. Unresolved: full re-harvest vs. delta from cursor. Confidence decay triggers re-validation before staleness becomes a problem.

📚 Non-API knowledge sources at scale

PDFs, spec sheets, image-based catalogs require different extraction strategies than REST APIs. Extraction pipeline: statistical NER → zero-shot NER → LLM fallback → merge & deduplicate (similar to Neo4j's entity extraction research).

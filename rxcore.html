How the grounding layer works

Three engineering disciplines work together to turn a general-purpose AI into a specialist in your product catalog and industrial domain.

Slide 3 · How It Works
🎛️
How it reasons
Orchestration
Purpose-built workflows — "find a substitute", "check compatibility", "configure a product" follow defined steps, not open-ended improvisation
Right workflow, every time — the agent selects the appropriate pattern at the start of each turn before doing anything else
Specialized sub-agents — complex queries are split across focused agents that each handle one piece of the problem in parallel
Built-in safety checks — inputs and outputs validated automatically; off-topic or unsafe requests are blocked before they reach the answer
🔌
What it knows
Knowledge Tools (from KCS)
Built from your catalog — queries structured tools generated from your own product data, not live brand APIs at question time
Understands relationships — what replaces what, what works with what, which families are compatible
Lifecycle-aware — always checks active vs. end-of-life status before recommending a product
Live inventory & pricing — stock levels and prices synced continuously so agents always quote accurately
🧠
What it remembers
Context Engineering
Three memory layers — long-term knowledge, the current conversation, and a fast cache for frequently accessed data
No stale distraction — old tool results are pruned so they don't mislead the model on later turns
Smart summarization — long conversations are compressed without losing key context from earlier in the session
Curated, not accumulated — every token in context is there because it's useful, not just because it was said
Reasoning · Knowledge · Memory

Before answering, RxCore checks everything at once

When a user asks a question, the agent fans out across your entire knowledge layer simultaneously, scores every piece of evidence, and only composes an answer once all research is complete — never before.

Slide 4 · Parallel Research & Confidence Scoring
Step 1
Parallel fan-out
All knowledge sources queried simultaneously:

• Knowledge graph — relationships & inferred compatibility
• Product database — official specs, lifecycle, declared replacements
• Semantic search — fuzzy natural-language matching
• Web fallback — fills gaps when internal data is incomplete
→
Step 2
Evidence scoring
Each result carries a confidence score:

• Graph: edge weight & path length
• Database: exact match vs. inferred join
• Semantic: similarity score & relevance
• Missing data returned as an explicit signal — never silently empty
→
Step 3
Reconcile & tier
Cross-source agreement sets the confidence tier:
HIGH
Graph + DB agree — stated plainly
MEDIUM
Single authoritative source — hedged
LOW
Semantic only — uncertainty disclosed
CONFLICT
Sources disagree — both surfaced
⚡
Why this produces better answers
No early commitment, no guessing
All research completes before any answer is written — the agent cannot short-circuit with a partial result
Multi-source triangulation eliminates the most common AI failure: anchoring on the first result found
Explicit "not found" signals trigger escalation to a broader search — never a fabricated answer to fill the gap
🎯
What you see as a result
Transparent, trustworthy answers
Every claim is sourced — the agent can show whether the answer came from official specs or inferred compatibility data
Uncertainty is surfaced, not hidden — if evidence is weak, the agent says so rather than sounding confident about a guess
Conflicts are shown — when sources disagree, both perspectives are surfaced so your team can decide
Parallel research · Scored evidence · Transparent answers

We train a specialist before they start the job

The Knowledge Construction System (KCS) pre-builds a structured, queryable knowledge layer from your catalog data — so the agent never scrambles to find information at the moment a customer asks a question.

Slide 5 · Knowledge Construction System
Input
Your data sources
• Files — PDF catalogs, datasheets, Excel specs
• Web — brand websites, product pages, documentation
• APIs — official brand APIs (e.g. Siemens SiePortal)
• Experts — your team contributes knowledge via chat
→
Process
KG Constructor agent
An AI agent that builds the knowledge layer:

• Ingestion — parses, normalizes, and deduplicates all sources
• Schema modeling — learns how this brand organizes products
• Validation loop — flags conflicts, proposes rules, queues changes for your team to review
→
Output
Structured knowledge layer
• Knowledge graph — product relationships, compatibility, replacement chains
• Product database — specs, lifecycle, stock, pricing from official sources
• Semantic index — enables natural-language product search
📈
Gets smarter with every brand
Brand-specific learning
Memory across onboardings — schema decisions and corrections from Siemens directly inform how ABB or Rockwell are processed next
Faster with each brand — what takes 20 iterations for brand 1 takes 5 for brand 10
Adapts to each catalog structure — Siemens, Rockwell, and Schneider organize products differently; KCS learns each
🔄
Stays up to date automatically
Continuous sync
Triggered by changes — new catalog files, API updates, or scheduled syncs kick off a targeted refresh
Low-risk changes auto-applied — price and stock updates go straight in; structural changes queue for review
High-frequency pricing & inventory — synced more often than taxonomy so agents always quote accurately
Build-time knowledge · Brand learning · Continuous sync

Three ways to answer any product question accurately

KCS produces three complementary knowledge surfaces. The grounding layer queries all three through stable, generated tools — it never calls brand APIs directly at the moment a question is asked.

Slide 6 · Knowledge Layer & Tools
🕸️
Knowledge Graph
Relationships & Inferred Compatibility
Captures the structure of your product catalog and the relationships your data implies — even when no single source explicitly states them. Powers "what replaces this?", "what works with that?", and cross-family recommendations.
graph_search get_taxonomy find_spare_parts recommend_modules
🗄️
Product Database
Official Source of Truth
What brand APIs and official sources explicitly declare: specs, certified replacements, lifecycle status, pricing, and stock. Kept separate from the graph so the agent can always be transparent about whether a claim is official or inferred.
sql_query get_stock_price get_alternatives check_compatibility
✨
Semantic Index
Natural Language Product Search
Embeddings generated from both the database and the knowledge graph. Handles vague, human-style queries like "something like the S7-1200 but cheaper" — surfaces candidates the agent then enriches with structured data.
rag_search
🧩
What onboarding a new brand looks like — Example: Siemens
Knowledge → Tools → Grounded Answers
Feed — SiePortal API and PDF catalogs are ingested by the KG Constructor
Build — a Siemens-specific knowledge graph is constructed; the product database is populated with official declarations from the API
Query — the grounding layer receives Siemens-specific tools and answers any question without ever calling SiePortal at query time
Graph · Database · Semantic — three surfaces, one agent

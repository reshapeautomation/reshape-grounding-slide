What is RxCore?

Not an agent — a platform. APIs, connections, knowledge grounding, memory, and data processes that power industrial automation intelligence. Consumed via API, tool, MCP, or A2A.

Slide 2 · Platform Overview
First-Party Apps
Chatbots, quote builders, bespoke UIs — add UI, business logic, prompting strategies, personalization, domain-specific behaviors on top of the core
External Integrations
CRMs, websites, email — route into the system through authentication and integration pathways
Grounding Layer
The agentic heart of RxCore — reasoning pipeline, memory, context, and search intelligence. Understands PLCs, SCADA, HMIs, sensors, drives, motor controls. Targets 99.9% accuracy.
RxCore Platform
Centralized intelligence & infrastructure — plug-and-play engine consumable via API, tool, MCP, or A2A. Everything above depends on it.
Knowledge Construction System
Builds the knowledge layer that RxCore queries — pre-constructed graph + RDBMS + embeddings, replacing fragile query-time integrations (see slide 4)

Grounding Layer · Three Pillars

The grounding layer rests on three engineering disciplines. Together they turn an LLM into an industrial-automation expert.

Slide 3 · Orchestration · Harness · Context

Orchestration Engineering
Behavior & Flow Control
Prompt engineering — system prompts, few-shot examples, instructions shape behavior
Skills — modular capabilities invoked per task (catalog search, spec lookup)
Multi-agents — specialized agents each handle a slice of the problem
Middleware — LangChain hooks for logging, guardrails, retries, PII
Custom workflows — LangGraph topologies for domain-specific reasoning

Harness Engineering
What The Agent Connects To
KCS-generated tools — stable pre-built interfaces replace fragile direct API calls
graph_search, sql_query, rag_search — three-way retrieval over the knowledge layer
get_alternatives, check_compatibility — replacement and fitment reasoning
get_stock_price, find_spare_parts — live inventory + accessory traversal
ERP, code exec, connections — business logic outside the knowledge layer
🧠
Context Engineering
Every Token Earns Its Place
StripToolHistoryMiddleware — removes stale tool pairs; fights distraction + poisoning
Memory manager — long-term, conversational, cache tiers
Message selection — System, Tool, Human, AI messages curated per call
Trim + summarize + prioritize — maximize signal within token budget
Ordering is deliberate — not every history message reaches the next call

Knowledge Construction System (KCS)

Shifts knowledge assembly from query time to build time. Fixes the fragility problem (APIs break, rate-limits hit) and the representation problem (PDF chunks have no structure).

Slide 4 · Build-Time Knowledge Assembly
Input
Data Sources
• Files — PDFs, Excel, images (webhook / upload / scheduled)
• Web — crawls, scrapes, structured extraction
• Domain — brand APIs (SiePortal), FDE chat, other agents
→
Agent
KG Constructor
• Ingestion pipeline — parse, normalize, dedupe, entity extract
• Schema modeling agent — iterates taxonomy, designs RDBMS, infers rules
• Validation + FDE review — flags conflicts, queues proposals
→
Output
Knowledge Layer
• Neo4j graph — inferred relationships, taxonomy
• RDBMS — source-of-truth declarations from APIs
• Embeddings — fuzzy semantic search over both
📚
Brand-Specific Learning
Expert-Per-Brand Over Time
Long-term memory — retains schema decisions, FDE corrections, extraction patterns per brand
Prompt learning — adapts to Siemens PDF tables, ABB API quirks, etc.
Agent optimization — 20 iterations for brand 1, 5 for brand 10
🔄
Job Sync
Living, Not One-Shot
Triggers — brand API updates, webhook file arrivals, scheduled freshness checks
Diff against graph — auto-apply low-risk changes (price, stock)
Queue structural changes — new families, changed relationships routed to FDE review
Query-time → build-time

Knowledge Layer → Tools

KCS produces three complementary query surfaces. Grounding layer consumes them through stable generated tools — never direct brand APIs at query time.

Slide 5 · Query Surfaces & Generated Tools
🕸️
Knowledge Graph
Neo4j · Inferred
Taxonomy of the full product catalog and relationships the data implies. Inferred compatibility, replacement chains, cross-family recommendations. "What makes sense from a data perspective."
graph_search get_taxonomy find_spare_parts recommend_modules
🗄️
RDBMS
Source of Truth · Declared
What APIs and official sources explicitly state. Specs, official replacements, lifecycle, stock, pricing. Stock + pricing sync at higher cadence than taxonomy.
sql_query get_stock_price get_alternatives check_compatibility
✨
Embeddings
Semantic · Fuzzy
Generated from RDBMS rows and graph nodes/edges. Handles vague natural-language intent — "something like the S7-1200 but cheaper" — surfaces candidates for graph/SQL enrichment.
rag_search
🧩
Example — Onboarding Siemens
Knowledge → Tools → Grounded Answers
1. Feed — SiePortal API + PDF catalogs feed the KG Constructor
2. Build — Siemens-specific taxonomy graph built; RDBMS populated with declared specs
3. Query — Grounding layer gets Siemens tools; never calls SiePortal at query time
Three surfaces · one agent
